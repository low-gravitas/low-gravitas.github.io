## Custom-domain ownership

- **`lowgravitas.com` CNAME owner:** `low-gravitas/low-gravitas.github.io` (the hub repo). Confirmed via `gh api repos/low-gravitas/low-gravitas.github.io/pages` — returned `"cname": "lowgravitas.com"`, `"build_type": "workflow"`, source `main/`, `"https_enforced": true`.
- **Pages routing model (important correction to plan v10 assumption):** Sibling repos under the same org are routed at `lowgravitas.com/<repo-name>/` via GitHub's org-level path routing *under the custom domain*, not on a separate `*.github.io` origin. The `html_url` field in each sibling's Pages API response confirms this: all three report `"html_url": "https://lowgravitas.com/<repo-name>/"`. This means Phase 5b/5e's post-disable verification should probe the apex URL (`lowgravitas.com/<repo>/`) and distinguish sibling-served vs. hub-redirect-stub content by inspecting response bodies, not by probing a separate origin.

## Deviations from plan v10 discovered during Phase −1

1. **Sibling Pages routes under custom domain, not separate origin.** Plan v10's wait+retry loops in Phase 5b step 11 and Phase 5e steps 23-24 probe `https://low-gravitas.github.io/<repo>/` expecting a 404 post-disable. That probe will 404 (or 301 to the apex) either way because sibling repo content lives at the apex path, not on `*.github.io`. **Adjusted verification** for Phase 5b/5e:
   - Before disable: `curl https://lowgravitas.com/<repo>/` returns sibling-served content (identifiable by characteristic HTML like `Low Gravitas Zen — Theme Preview` title or the font demo's hero icons).
   - After disable: same URL returns the hub's redirect-stub HTML (identifiable by `<meta http-equiv="refresh" content="0;url=/zen-theme/">`).
   - Distinguish by `grep -q 'meta http-equiv="refresh"'` on the response body, not by HTTP status code.

2. **RSS feed self-link bug pre-exists consolidation.** Current `https://lowgravitas.com/blog/feed.xml` contains `<link href="https://lowgravitas.com/feed.xml" rel="self"/>` (wrong path). Fix in Phase 2b during the hardcoded URL cleanup — update the blog's `feed.njk` template to correctly reference `/blog/feed.xml` as self. This also means `sitemap.njk` in Phase 1 should NOT try to emit `/feed.xml` (there is no such path in production today, despite what the feed XML claims).

3. **Blog repo `https_enforced: false`.** Pre-cutover optional fix: enable HTTPS enforcement in blog repo Settings → Pages to match sibling-repo consistency, or accept the anomaly since the consolidation will retire blog Pages anyway in Phase 5b.
